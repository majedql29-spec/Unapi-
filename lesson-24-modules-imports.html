Lesson 24: Modules & Imports

Advanced⏰ 8 min

Python's standard library is vast — hundreds of modules covering mathematics, networking, file I/O, data processing, web development, and more. The import statement is how you access this power and organize your own code into reusable units. Understanding import styles and module structure is essential for writing maintainable Python projects at any scale.

Import styles and when to use each

  • import math — imports the entire module. Access with math.sqrt(). Best when you use many functions from a module — the prefix prevents naming collisions and makes the source clear.
  • from math import sqrt, pi — imports specific names directly. Best when you use only a few items and the module name would add unnecessary verbosity.
  • from math import * — imports all public names. Avoid this — it pollutes your namespace, can silently override existing names, and makes it impossible to tell where a function comes from.
  • import math as m — imports with an alias. Use for modules with long names: import numpy as np; import pandas as pd — these are community conventions you should follow.
import math
from random import randint

print(f"Pi: {math.pi}")
print(f"Random 1-10: {randint(1, 10)}")

The __name__ == "__main__" pattern

This allows a file to be both used as a reusable module and run as a standalone script:

def main():
    print("Running as script...")

if __name__ == "__main__":
    main()

When you run a Python file directly (python myfile.py), Python sets __name__ to "__main__". When the file is imported as a module, __name__ is set to the module's name. This pattern lets you put test code or CLI logic inside the guard so it doesn't run when the module is imported elsewhere.

Creating your own modules and packages

Any .py file is a module. A package is a directory of modules with an __init__.py file. Structure:

my_project/
  __init__.py
  utils/
    __init__.py
    strings.py
    files.py
  main.py

Then from my_project.utils.strings import capitalize or from my_project import utils. The __init__.py can control what gets exported via __all__ or run initialization code.

sys.path and how Python finds modules

When you import a module, Python searches the list of directories in sys.path: (1) the directory of the script being run, (2) directories listed in the PYTHONPATH environment variable, (3) standard library directories, (4) site-packages (for third-party packages installed via pip).

Common mistakes

Circular imports — module A imports module B which imports module A. Restructure code or use lazy imports inside functions. Using from x import * in production code. Not organizing imports — causing confusion about where functions come from. Putting executable code (not inside a function or guard) in a module that's meant to be imported.

Best practices

Organize imports in three groups, separated by blank lines: standard library, third-party, local modules. Within each group, sort alphabetically. Use absolute imports (from package.module import name) rather than relative imports (from ..module import name). Use if __name__ == "__main__": guard for all script files.

Real-world use case

A data analysis project might import: import os, sys (stdlib), import pandas as pd, import numpy as np, import matplotlib.pyplot as plt (third-party), from myproject.loaders import load_data; from myproject.analysis import calculate_stats (local). This three-group organization makes dependencies immediately visible and enables reuse across different scripts.

Try it: Create a module in your mind: write a function in the editor, then imagine putting it in a separate file utils.py and importing it. Then use the if __name__ == "__main__": pattern to add test code that only runs when the file is executed directly. Finally, import a module with an alias: import datetime as dt and use dt.datetime.now().
