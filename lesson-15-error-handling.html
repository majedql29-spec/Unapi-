Lesson 15: Error Handling

Intermediate⏰ 7 min

Errors happen — users enter invalid input, files are missing, networks fail, data is corrupted. Instead of letting your program crash with an ugly traceback, you can handle exceptions gracefully using try/except blocks. Python's exception handling is both powerful and flexible, allowing you to respond to errors in fine-grained ways.

try:
    n = int(input("Enter a number: "))
    print(10 / n)
except ValueError:
    print("That's not a number!")
except ZeroDivisionError:
    print("Can't divide by zero!")
else:
    print("No errors!")
finally:
    print("Done")

The complete try/except/else/finally pattern:

  • try — wrap code that might raise an exception here. If an error occurs, the rest of the try block is skipped immediately.
  • except SomeError — catches a specific exception type. You can have multiple except blocks for different error types. Use except Exception as e: to capture the exception object and inspect its message (str(e)). A bare except: catches everything, including SystemExit and KeyboardInterrupt — avoid it or use except Exception: instead.
  • else — runs only if no exception occurred in the try block. This is where you put "success path" code. It's better than putting it in the try block because it won't accidentally catch exceptions from unrelated code.
  • finally — always runs, whether an exception happened or not. Ideal for cleanup: closing files, releasing locks, closing network connections, restoring state.

Exception hierarchy: Python exceptions are organized in a class hierarchy. BaseException is the root (don't catch this). Exception is the base for all built-in, non-system exceptions. Common subclasses include ValueError (wrong value), TypeError (wrong type), IndexError (list index out of range), KeyError (missing dict key), FileNotFoundError (missing file), ZeroDivisionError (division by zero), AttributeError (missing attribute), and ImportError (missing module). You can catch a parent type to handle a family of related exceptions: except (ValueError, TypeError):.

Raising exceptions — sometimes you need to signal an error yourself: raise ValueError("Invalid input"). You can also re-raise the current exception inside an except block with bare raise (preserving the original traceback). Create custom exceptions by inheriting from Exception:

class InsufficientFundsError(Exception):
    """Raised when account balance is too low."""
    pass

Then raise and catch it like any built-in exception. Custom exceptions make your code's error handling more expressive and domain-specific.

Common mistakes: Catching too broadly (except: or except Exception: when you only expect one type) — this can hide bugs. Not ordering except blocks from most-specific to least-specific (Python checks them in order, so a generic handler before a specific one will catch everything). Using exceptions for normal control flow — they should be for exceptional situations, not regular program logic. Swallowing exceptions silently with except: pass — at minimum log the error.

Best practices: Catch only exceptions you can actually handle. Let unexpected exceptions propagate to the top level where they can be logged or shown to the user. Be as specific as possible about which exceptions you catch. Use else for success-path code to separate it from error-prone code in the try block. Use finally for cleanup that must happen regardless of success or failure. When raising exceptions, provide informative messages that help with debugging.

Real-world use case: When processing user uploaded files, you might wrap the entire process: try: data = parse_file(path) — catching FileNotFoundError (file missing), PermissionError (no access), json.JSONDecodeError (malformed content), and UnicodeDecodeError (wrong encoding) — each with a different user-friendly error message, plus a finally block to clean up temporary files.

Try it: Add an else block to the code that prints a success confirmation. Add a finally that prints "Operation complete". Then create a custom exception class and raise it conditionally. Finally, try catching multiple exception types with a tuple: except (ValueError, ZeroDivisionError) as e:.
