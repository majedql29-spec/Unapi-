Lesson 17: Lambda Functions

Intermediate⏰ 7 min

A lambda is a small, anonymous function defined with the lambda keyword. Unlike regular functions defined with def, lambdas are written in a single expression, have no name, and cannot contain statements or annotations. Syntax: lambda arguments: expression. The expression is evaluated and returned automatically — no return keyword needed: lambda x, y: x + y creates a function that adds two numbers.

# Lambda = anonymous function
square = lambda x: x ** 2
print(square(5))

add = lambda a, b: a + b
print(add(3, 7))

# Use with sorted
points = [(1, 5), (3, 2), (2, 8)]
points.sort(key=lambda p: p[1])
print("Sorted by y:", points)

Lambdas shine when you need a simple, throwaway function, especially as an argument to higher-order functions:

  • sorted() with a custom key: students.sort(key=lambda s: s["grade"]) — sort a list of dicts by the "grade" field. Also works with max() and min(): max(points, key=lambda p: p[1]) finds the tuple with the largest y-coordinate.
  • map(): list(map(lambda x: x * 1.15, prices)) — apply a 15% tax to each price.
  • filter(): list(filter(lambda x: len(x) > 5, words)) — keep only words longer than 5 characters.

When NOT to use lambda: If the logic requires statements (if/elif/else chains, loops, try/except, variable assignments), you cannot use lambda — use def. If the function is used in multiple places, define it once with def to avoid repetition. If the lambda expression is too long to read comfortably on one line, use def. As a rule of thumb: if you need more than a single expression or the expression is longer than ~80 characters, write a regular function.

Lambda vs def comparison: Both create function objects that can be called the same way. def functions have a name (helpful in tracebacks), support docstrings, annotations, and multiple statements. Lambdas are limited to a single expression and have no name or docstring — if they raise an error, the traceback shows <lambda> which is less helpful for debugging. Performance-wise, they are identical — choice is purely about readability and capability.

Common mistakes: Trying to use statements inside a lambda — lambda x: return x + 1 is a SyntaxError. Using a lambda when functools.partial or a regular function would be clearer. Assigning a lambda to a variable (add = lambda x, y: x + y) is poor style — if you're going to name it, use def instead. That's what def is for.

Best practices: Use lambdas only for simple, single-use operations. Prefer list comprehensions over map() with lambdas for readability. Use key= arguments in sorted(), max(), min() — these are the most common and idiomatic use cases. If the lambda body is complex, refactor it into a named function with a descriptive name.

Real-world use case: Sorting a list of product objects by price descending: products.sort(key=lambda p: p["price"], reverse=True). Finding the employee with the highest salary: best = max(employees, key=lambda e: e.salary). Grouping data with itertools.groupby: for city, group in groupby(people, key=lambda p: p["city"]):.

Try it: Use max() with a lambda to find the longest word in a list: words = ["cat", "elephant", "dog"]; print(max(words, key=lambda w: len(w))). Then use sorted() with a lambda that sorts by the last character of each word. Finally, compare the readability of a lambda vs a def for a multi-step key function.
