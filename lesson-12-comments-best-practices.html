Lesson 12: Comments & Best Practices

Beginner⏰ 5 min

Comments exist to explain code to human readers — including your future self who will come back to this code six months from now. Writing good comments is a skill that separates professional developers from beginners. In Python, use # for single-line comments and triple-quoted strings (""" """) for multi-line docstrings. A good comment explains why something is done a certain way, not what the code does — the code itself should be clear enough to communicate what it does without comments.

Types of comments in Python:

  • Inline comments (#) — explain a specific line or block. Use sparingly and only when the code is not self-explanatory.
  • Docstrings (""" """) — document functions, classes, and modules. These can be accessed programmatically via help() or __doc__, making them essential for creating usable libraries.
  • TODO comments (# TODO: fix this later) — mark unfinished work. Many IDEs highlight these for easy navigation.

Best practices for writing clean, professional Python:

  • Use meaningful names — user_age is far better than ua or x. Good names eliminate the need for many comments.
  • Follow PEP 8 — Python's official style guide. Key rules: 4-space indentation (not tabs), 2 blank lines around top-level functions, 1 blank line around methods, lowercase_with_underscores for variables and functions, UPPER_CASE for constants.
  • Keep functions small — each function should do exactly one thing and do it well. If a function is doing multiple things, split it. Small functions are easier to test, debug, and reuse.
  • Avoid magic numbers — never scatter raw numbers like 86400 (seconds in a day) throughout your code. Instead, define a named constant: SECONDS_IN_DAY = 86400. This makes the code self-documenting and easy to change in one place.
  • Write docstrings for every public function, class, and module — they enable auto-generated documentation and help users understand how to use your code without reading the implementation.
  • DRY (Don't Repeat Yourself) — if you find yourself copy-pasting code, extract it into a function or variable. Repeated code is a breeding ground for bugs when you update one instance but forget to update the others.
  • Keep imports at the top — standard library imports first, then third-party, then your own modules, each group separated by a blank line.

Common mistakes in commenting:

  • Writing obvious comments: x = x + 1 # increment x by 1 — the code already says this. Explain why, not what.
  • Letting comments rot: When you update code, update the comments too. Outdated comments are worse than no comments — they actively mislead readers.
  • Using comments as a substitute for clean code: Instead of commenting confusing logic, rewrite the logic to be clearer.

Remember the golden rule of Python: Readability counts. Code is read far more often than it is written. Spend the extra few seconds to make your code clear — your future self and your teammates will thank you.
